Crowd-Sourcing Data Analysis

This document is a system specification and best practices guideline for importing crowd sourcing data that is presumably collected from related SDKs via mobile apps for the purpose of network performance measurement and benchmarking. Methods, algorithms and thresholds are derived from experience and subject to change as the dynamics of the mobile network of interest changes.

Target readers are analysts, software engineers and database designers.

Note: This document assumes that interested reader knows what crowd sourcing is and the basic principles of how crowd-sourced data is collected via applications and what type of data is collected.

For the rest of the document sample data from Ookla SpeedTest web site will be used as reference.

3.2. An algorithm to use a Web Service

Again as mentioned above we assume Ookla Data as an example, but the following algorithm can be easily modified to use any web service that gathers raw data of any format. Here it must be noted that although a web service is available its purpose is to download separate files with specific name format.

A more common web service that directly gathers data in formats like JSON, XML may already have a parametric syntax and would make all the mess below much simpler :)

There are 3 different types of extracts depending on the operating system other terminal doing the SpeedTest:

  • Android extracts: The most detailed and biggest type of extracts
  • IOS extracts: Come from apple terminals and contain relatively less information and relatively less number of lines
  • Windows extracts: speed test data coming from windows terminals. Contains much less number of lines. We prefer not process these extracts for practical purposes.

As can be seen from the sample data, each row in these extracts correspond to one and only one measurement, hence independent. This fact is very important especially in reporting phase. Some crowd sourcing provide aggregated data usually over geographical and date-time dimensions and that requires a different design and handling which we will not go into in this document.

Some of the columns are different for different types, details of which will be given in later chapters.

4.1. Parsing Module

The main purpose of the parsing module is:

  • Correct unrecognized characters
  • Avoid wrong column shifting due to comas inside columns
  • Bring android and ios extracts to an equivalent format, so that they can be inserted to the same table in final database.
4.1.1. Row Corrections

Row corrections are set of rules to be applied to each and every row of data. These rules are independent of the extract type. i.e. same rules are applied to both android and ios extracts. List of rules to be applied to each row is as follows:

  • Remove ‘?’s
  • Remove successive 16 spaces ‘ ‘
  • Replace every character to its equivalent in ISO Basic Latin Alphabet as capital letters.
  • Split the row in correct number of columns using coma (‘,’) as delimiter.

ISO Basic Latin Alphabet as capital letters

4.1.2. Column Corrections

Once the row corrections are done and row is split properly (i.e. has correct number of columns) there may be necessary corrections that must be applied to each specific column. These corrections can be different for ios and android. The most important step after column corrections is mapping the column to an equivalent column for both ios and android. 5. DATA ANALYSIS

This chapter assumes a database equivalent to the final database created according to chapters 2 and 3. There are almost infinite different possibilities of querying the final database for different purposes. This document does not intend to force certain data-mining rules but following chapters are derived from experience and should be considered as best practice methods.

5.1 LTE Considerations

LTE throughputs (especially downlink), can spread from a few megabits to hundreds of megabits. Although several factors like frequency usage, terminal category etc. can directly affect the results, typical LTE downlink throughput histogram tends to have a Rayleigh Distribution character. It must be noted that crowdsourcing measurements are never going to be %100 clean. i.e. due to uncontrolled environment and millions of different terminals some theoretically impossible values can be observed in the extracts. This means the results must be properly filtered before starting any analysis.

Here, note that a value of one single test with 1 Gbps DL speed would affect the average throughput of 10000 users 0.1 Mbps. This becomes more and more important as the number of users increase and less important as users decrease. Therefore, if we select the upper limit of acceptable speeds too high, a wrong value may affect the average throughput calculation (even the probability of encountering them is % 0.01 for the example above). The story is more or less same for uplink.

Keeping the below issues in mind, for the case of LTE, following filters can be used as a rule of thumb. However it must be noted that these values are subject to change due to the latest technological improvements in LTE networks and terminal capabilities.

KPI column Less Than
[download_kbps] 500.000 Kbps
[upload_kbps] 60.000 Kbps
[latency] 500 ms

It must be noted that these values are the best choices at the time I prepared this document. LTE technology keeps evolving and theoretical maximum speeds have to be taken into account.

Another important point on filtering LTE data is the [connection_type] column. Note that Table 1: android connection type mapping has LTE and LTE_CA (carrier aggregation) and Table 2 ios connection type mapping has LTE connection types. That is, a filtering of LTE tests must include both LTE, and LTE_CA.

By the time I decided to publish this document, several other connection technologies can be detected by SDKs. So interested reader must be careful that all LTE related connection types are added.

5.2. 3G Considerations

For 3G (including HSPA), a very similar scenario is valid. Following filter values can be used as a rule of thumb.

KPI column Less Than
[download_kbps] 84.000 Kbps
[upload_kbps] 20.000 Kbps
[latency] 500 ms

It must be noted that these values are the best choices at the time I prepared this document. Even HSPA technology keeps evolving and theoretical maximum speeds have to be taken into account.

5.3. AGGREGATION CONSIDERATIONS

Crowd-sourced data is almost always uncontrolled data. i.e. it is not possible to control who, where, when and how many times a measurement is made. My experience suggests that, biased data can be encountered. Cases where some specific user did excessive number of tests under specific conditions with specific type of phone can lead to deviation of actual results especially if the number of this biased test is significant compared to the total number of tests in a data set.

In order to minimize this effects from the field, understanding the following 3 figures is important.

5.3.1. TEST COUNT

Test count is total number of tests in a dataset. This figure can be calculated by simply counting the values in [test_id] column (which we know are unique at anytime, anywhere in the whole db).

5.3.2. DEVICE COUNT

Device count is the total number of distinct devices in a dataset. This figure can be calculated by simply counting the distinct values in [device_id] column.

5.3.3. SAMPLE COUNT

Consider the following scenario:

We want to calculate weekly downlink throughput trend of “an operator” in “a city” for the “last N weeks”. We also know that there has been an excessive test attempts by one specific user who happens to measure “0 kbps” dl tput due to an issue which is out of scope of this document.

  • If we simply take the average of all downlink speeds in the dataset of interest, we will encounter an unnatural decrease in trend for the last week due to that one user who has an issue and did several measurements.
  • If we group the results by distinct devices first, then we average that user as if he only took one test for the whole week.

Sample Count is a combination of this 2 extreme cases and is defined as follows:

Number of data points that is created by one user in one day in a given dataset. i.e. tests taken by one specific user is grouped by [device_id] and [test_day] Our experience suggests that any aggregation to derive any given dataset must not start from tests, or devices but samples, definition of which is straightforward given the sample count definition above.

5.3.4. GEOGRAPHICAL CONSIDERATIONS

One of the most intuitive analysis cases of looking into crowdsourcing data is to monitor the performances of different cities/regions etc.

Before attempting such an analysis, remember that our data is being collected via SDKs embedded in mobile applications. In either case, quality of data is directly related to how spread and popular the application is.

There may be applications:

  • used by customers only on a specific city
  • used by any specific segment of customers (i.e. age, occupation, sex etc.)
  • used only at specific times

For example, in countries like Turkey, number of tests is proportional to population, which means we will have non-uniform distribution amongst different cities. One of the easiest mistakes is to attempt to calculate average value of a kpi for relatively small cities and compare them.

My experience suggests that, the following thresholds must be satisfied at minimum in order to be able to talk about a meaningful benchmarking amongst cities or any other geographical set.

Period rules
Week Nr of distinct users > 10 per week
Month Nr of distinct users > 30 per month
Quarter Nr of distinct users > 100 per quarter
